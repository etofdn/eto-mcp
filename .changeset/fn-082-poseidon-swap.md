# FN-082: Poseidon-2 swap in eto-zk (T-1.3.1.4)

## Summary

Replaces the v0 polynomial scaffold hash functions in `crates/eto-zk` with
production **Poseidon-2 over BN254 `Fr`, `t=3`**, per the FN-212-ratified
parameter set in `spec/SINGULARITY-LAYER-1.md` ┬¦10.3.1.

## Changes

### (a) `commit_to_value` and `merkle_compress` now use Poseidon-2 t=3 over BN254 `Fr`

The v0 scaffold functions `native_claim_commitment` and `native_merkle_compress`
in `crates/eto-zk/src/circuits/credential_predicate.rs` now delegate to the new
`crates/eto-zk/src/poseidon.rs` module:

- `native_claim_commitment(value, blinding)` ŌåÆ `poseidon::commit(value, blinding, 0)`
  ŌĆö `Poseidon2_t3([value, blinding, Fr::from(0)])`
- `native_merkle_compress(left, right)` ŌåÆ `poseidon::compress(left, right)`
  ŌĆö `Poseidon2_t3([left, right, DOMAIN_MERKLE])`

The in-circuit gadgets (inside `ConstraintSynthesizer`) also use `PoseidonSpongeVar`
(from `ark-crypto-primitives 0.5`) instead of the polynomial formulas. The native
and in-circuit implementations are now guaranteed to agree because they share the
same parameter singleton (`poseidon::parameters()`).

**Parameters**: BN254 `Fr`, state width `t=3` (rate=2, capacity=1), 8 full rounds,
56 partial rounds, S-box x^5, MDS + ARK constants generated by the Grain LFSR
(`find_poseidon_ark_and_mds` from `ark-crypto-primitives 0.5`).

### (b) `eto-zk-cli commit` ŌĆö canonical issuer helper for `claimCommitments`

A new top-level `commit` subcommand has been added to `eto-zk-cli`:

```text
eto-zk-cli commit --field <name> --value <v> --idx <i> --salt <hex32>
```

Output (matches ┬¦10.3.1 per-entry JSON shape):

```json
{"fieldPath": "<field>", "idx": N, "commitment": "<64 hex>", "saltCommitment": "<64 hex>"}
```

- `commitment` is 32-byte little-endian hex of `Poseidon2_t3([value_field, salt_field, idx_field])`
- `saltCommitment` is 32-byte little-endian hex of `Poseidon2_t3([salt_field, Fr::ZERO, idx_field])`
- Salt is decoded via `Fr::from_le_bytes_mod_order` (little-endian, per ┬¦10.3.1)
- Value encoding supports auto-detection + explicit `--encoding {auto,numeric,string,bool}`

### (c) On-chain verifier VK must be regenerated

The `verify_credential_predicate` circuit's R1CS shape has changed (from ~1 363 to
~9 454 constraints). The embedded proving/verifying keys stored in the on-chain
`Groth16VerifyingKey` are now **invalidated**. The `EMBEDDED_VERIFYING_KEY` constant
in `svm-runtime` **MUST be regenerated** in a follow-up task before the new circuit
is deployed to devnet or mainnet.

**Follow-up task**: FN-008.

### (d) Pre-FN-082 `claim_commitment` test fixtures are invalidated

Any test fixture or integration test that hardcodes a `claim_commitment` value
computed under the v0 scaffold (polynomial hash) is no longer valid. Regenerate
fixtures using:

```bash
eto-zk-cli commit --field <fieldPath> --value <value> --idx <idx> --salt <hex32>
# or in Rust:
eto_zk::poseidon_commit(value_field, salt_field, idx)
```

No such hardcoded fixtures were found in the workspace at the time of this PR.

## Downstream impact

- `eto-zk` crate: re-exported `poseidon_commit`, `poseidon_salt_commit`, `poseidon_compress`
- `eto-zk-cli` binary: new `commit` subcommand
- `svm-runtime` verifier: VK regeneration required before on-chain deployment
